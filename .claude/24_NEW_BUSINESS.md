# 24. 신규 개업(인허가) — 관리자 확인용

> "새로 개업하면 어떻게 알고 영업 전화가 오나?" 를 조사해, 같은 데이터를 **합법적으로** 받아 관리자에서 열람하고
> 해당 업소의 **네이버 플레이스 등록 여부**까지 붙인다. (2026-07-17 구현·검증 완료)

## 왜 이게 되나 — 영업 전화의 출처

네이버가 흘리는 게 아니다. **지방행정 인허가 공공데이터**가 원천이다. 음식점·미용업 등은 개업 시 지자체 인허가를 받고, 그 내역이 매일 공개된다(상호·주소·인허가일자·**소재지전화**·업태·좌표). 여기에 이를 재가공해 파는 **상용 "신규 사업자 DB 추출기"** 시장(사주모 39.9만원, 마케팅몬스터 월 25만원 등)이 존재한다 — 다만 벤더의 "국세청 자료" 주장은 미검증이고, **판매된다는 사실이 적법의 근거는 아니다**.

- **국세청 사업자등록 API 는 쓸 수 없다** — 사업자번호를 이미 알아야 하는 **조회형**이고 응답에 상호·주소·전화가 없다. 과세정보는 국세기본법 제81조의13상 원칙적 비공개(법제처 11-0345)라 신규 명단이 공개될 구조가 아니다. (확보한 번호의 휴·폐업 **검증**엔 유효)
- **네이버 플레이스 diff 수집은 보류** — 야놀자 v 여기어때(대법원 2021도1533)는 형사 무죄지만 **같은 행위에 민사 10억 배상**(부정경쟁방지법 성과도용)이 인정됐고, 그 무죄 근거가 "피해자 무보호조치"라 로그인·nCaptcha 가 있는 네이버엔 원용 못 한다. 저작권법 제93조 2항은 반복·체계적 수집을 '상당한 부분 복제'로 간주한다.

## ⚠️ 법적 안전선 (이 기능의 설계 제약)

- **문자·이메일 광고는 명시적 사전 동의 필수**(정보통신망법 제50조 1항). 공공데이터로 얻은 번호로 보내면 원칙적 위법이고 **수신자가 사업자여도 면제 없다**. 과태료 **750 → 1,500 → 3,000만원**. "공개된 번호면 괜찮다"는 실무상 오해.
- 육성 전화는 법정 예외지만 ①방문판매법상 전화권유판매업 **신고** ②육성(ARS 불가) ③**수집출처 고지** 3요건을 모두 지켜야 한다.
- **따라서 발송(문자·메일) 기능을 붙이지 말 것** — 코드 주석과 이 문서에 못 박았다. (화면 상단 고지 배너와 자동 파기는 2026-07-17 운영 요청으로 제거)
- 개인사업자의 상호·주소·전화는 개인정보일 수 있는 회색지대 → 전화는 암호화 저장, 열람 로그·수집 출처 기록은 유지.

## 1단계 — 인허가 수집

- 소스: **서울 열린데이터광장** 인허가(LOCALDATA 서울 제공분). ⚠️ 전국 포털 `localdata.go.kr` 은 2026-04 폐쇄로 **응답하지 않는다**(실측). 전국 확장은 data.go.kr 표준데이터 인증키 발급 후 별도 클라이언트.
- [SeoulLocalDataClient](../app/Domain/NewBiz/SeoulLocalDataClient.php) — `GET {base}/{KEY}/json/{서비스}/{start}/{end}/{인허가일자}`
  - **첫 선택인자 = 인허가일자 필터**(실측): `2026-07-10` → 48건 · `2026-07`(접두) → 483건 → **전체 53만건 스캔 불필요, 하루치 1콜**
  - ⚠️ **`decode_content=false` 필수** — Guzzle 기본값이면 `cURL error 61: Unrecognized content encoding type` 로 전량 실패(실측). 헤더로 identity 를 줘도 안 되고 이 옵션이라야 한다.
  - ⚠️ 오류가 **XML** 로 오기도 한다(sample 키 범위 초과 `ERROR-335`) → JSON 파싱 전에 처리.
  - ⚠️ `sample` 키는 **1~5 범위만** 허용 → 페이지네이션 금지·일자당 5건. 정식 키는 data.seoul.go.kr 발급 → `.env SEOUL_OPENAPI_KEY`
- [NewBusinessCollector](../app/Domain/NewBiz/NewBusinessCollector.php) — 관리번호(MGTNO) 기준 upsert + 주소 → 시/도·시/군/구·읍면동 파생
- ℹ️ **전화번호(SITETEL)는 원천에서 대부분 비어 있다** — 파싱 문제가 아니다(실측 2026-07-17: 2026-07-15 인허가분 20건 중 5건만, 2015-07-15 인허가분도 6/20). 인허가 신고서에 전화 기재가 필수가 아니라 원천 자체가 결측이다. 우리 DB 254건 중 236건 NULL 이 정상 → **플레이스 등록 번호로 보완한다**(아래 [PlacePhoneFetcher](../app/Domain/NewBiz/PlacePhoneFetcher.php)).
  - ⚠️ **날짜 필터는 서비스마다 동작이 다르다**(실측): 일반음식점(072404)은 먹지만 **휴게음식점(072405)은 무시돼 전체 14.6만건(2001년 폐업 포함)이 그대로 온다** → 실제로 오염 적재가 발생했다. 그래서 수집기가 **응답 행의 인허가일자를 요청 날짜와 대조해 어긋나면 버리고 오류를 낸다**. 새 업종 추가 전 반드시 필터 동작 확인(`config rankfree.newbiz.services` 주석 참조).
- 커맨드: `newbiz:collect {--date=} {--days=} {--svc=} {--no-place}` — 원천이 **D-2 기준 현행화**라 오늘·어제는 비어 있는 게 정상(기본이 D-2 부터 조회).
  - **수집과 플레이스 확인은 한 흐름이다**(2026-07-17 운영 요청 — "따로 도는 게 아니고"). 수집이 끝나면 같은 실행에서 확인 대상(`scopeNeedsPlaceCheck`)을 **상한 없이 전부** 매칭한다(`--no-place` 로 수집만 가능).
  - 인증키는 [환경 설정 > 연동 > 서울 열린데이터광장](../resources/views/admin/settings/index.blade.php)(암호화 저장 → `SettingsServiceProvider` 가 `rankfree.newbiz.seoul_key` 오버라이드).

## 2단계 — 네이버 플레이스 매칭

- [NaverLocalSearchService](../app/Domain/NewBiz/NaverLocalSearchService.php) — **공식** 지역검색 OpenAPI(`openapi.naver.com/v1/search/local.json`), 쇼핑과 같은 개발자센터 키 재사용.
- ⚠️ **pcmap 순위검색(PlaceRankChecker)으로 하면 안 된다** — 상호 검색이 안 된다(실측: `신천동 맛집` 11,758건 정상 / `조선솥밥` 0건). 처음 이 방식으로 만들었다가 **전부 '미등록' 오판**이 났다.
- [NewBusinessPlaceMatcher](../app/Domain/NewBiz/NewBusinessPlaceMatcher.php) — "{동} {상호}" 검색 → 0건이면 상호만으로 재시도. 판정: **상호 정규화 일치**(공백·특수문자 제거 후 포함 관계 — "풍납 주먹고기 풍납본점" ~ "풍납주먹고기") **AND 같은 시/군/구**(다른 지역 동일 상호 오탐 방지).
- ⚠️ 공식 API 응답에 **플레이스 ID 가 없다** → 링크는 `map.naver.com/p/search/{상호} {지역}`(클릭 시 해당 플레이스로 이동). m.place 상세 URL 은 만들 수 없다.
- 커맨드: `newbiz:place-match {--limit=} {--recheck}` — 기본이 **제한 없음**(대상 전부). `--recheck` 는 주기와 무관하게 영업 중 전 업소 재조회. 평상시엔 `newbiz:collect` 만으로 매칭까지 끝난다.
- **미등록은 끝이 아니다 — 자동 재확인**([scopeNeedsPlaceCheck](../app/Models/NewBusiness.php)): 개업 직후엔 플레이스가 없다가 며칠~몇 주 뒤 여는 게 흔하다(2026-07-17 운영 요청). 확인 대상 = `pending` + **`recheck_after_days`(기본 3일)이 지난 `not_found`·`blocked`**, 단 인허가 후 `recheck_max_age_days`(기본 90일)까지만(그 뒤로도 없으면 플레이스를 안 내는 업소로 본다). 재확인에서 플레이스가 잡히면 `found` 로 바뀐다 — 실측: 미등록이던 12건을 재확인 대상으로 되돌려 전량 재조회 확인.
- **주기를 안 기다리는 '전체 재확인'**([scopePlaceCheckTarget](../app/Models/NewBusiness.php) + `force=1`): 방금 확인해 대상이 0 이어도 관리자가 지금 다시 보고 싶을 때가 있다(2026-07-17 — "미등록 업체 확인 안 되는데"). 주기를 무시하고 미확인·미등록 **전부** 재조회한다. 배치 루프가 끝나도록 서버가 **실행 시작 시각(`since`)** 을 잡아 응답에 실어주고 화면이 이후 배치마다 되돌려 보낸다 — 방금 처리한 건(`place_checked_at >= since`)은 대상에서 빠진다. ⚠️ `since` 는 **초 단위**라 실행 시작과 같은 초에 확인된 건은 제외된다(방금 본 건이므로 의도된 동작). 실측: 미등록 194건 전량 재확인.
- **건수 상한은 없다.** `place_match_batch`(기본 5)는 상한이 아니라 **화면 AJAX 1요청당 배치 크기**다 — 한 요청에서 수백 건 × 네이버 왕복이면 게이트웨이 타임아웃이라 끊어서 이어 부르고, 작게 잡아야 진행률이 자주 갱신된다. CLI 는 배치 없이 한 번에 전부.

## 2단계-b — 플레이스 등록 전화번호([PlacePhoneFetcher](../app/Domain/NewBiz/PlacePhoneFetcher.php))

인허가 원천의 전화가 대부분 결측이라, **플레이스가 있고(`found`) 인허가 전화가 없는 업소만** 번호를 가져온다(2026-07-17 요청). 매칭 성공 시 [NewBusinessPlaceMatcher::save()](../app/Domain/NewBiz/NewBusinessPlaceMatcher.php) 에서 호출 — 수집과 한 흐름.

- **경로(실측 2026-07-17 — 브라우저·토큰·쿠키 없이 PHP curl 로 전부 가능)**:
  1. `pcmap.place.naver.com/place/list?query={지역 상호}` HTML → `window.__APOLLO_STATE__` 의 **`PlaceListBusinessesItem:{id}`** 에 id·상호·주소·`phone`·`virtualPhone`·`hasMobilePhoneNumber` 가 **한 요청에 다 온다**(파싱 정규식은 기존 `PlaceRankChecker::parseDetailHtml` 과 동일).
  2. `phone`(일반) → `virtualPhone`(안심번호 0507…) 순으로 채택.
  3. 둘 다 null 인데 `hasMobilePhoneNumber` 면 → `pcmap-api.place.naver.com/rest/phone?id={id}` 가 **base64** 로 `{"number":"010-…"}`.
- ⚠️ **rest/phone 하나로는 안 된다**(실측): 풍납주먹고기(1965864491)는 virtualPhone=0507-1314-8662 인데 rest/phone **404** / 메구미카페(2062089507)는 phone·virtualPhone 둘 다 null 인데 rest/phone=010-9636-2469. **둘은 상호 보완 관계**다.
- ⚠️ **pcmap GraphQL 순위 API 는 상호 검색이 안 되지만(2단계 주석), 검색 페이지 HTML(`place/list`)은 상호로도 찾힌다** — 여기서 쓰는 건 후자. 공식 지역검색 OpenAPI 는 `telephone` 을 **빈 문자열로** 내려주므로(실측) 쓸 수 없다.
- 저장: `place_id`(그동안 비어 있던 컬럼이 이제 채워진다) · `place_phone`(**encrypted**) · `place_phone_type`(`virtual`|`normal`). 표기는 [NewBusiness::displayTel()/telSource()](../app/Models/NewBusiness.php) — 인허가 번호 우선, 없으면 플레이스 번호 + `안심`/`플레이스` 라벨.
- ⚠️ 안심번호(0507)는 **업체 실번호가 아니다**(네이버 중계번호) — 화면에 라벨로 구분해 표기한다. 실측상 신규 개업은 대부분 안심번호다(5/5).
- ⚠️ 수집한 번호도 **광고 발송 금지**(정보통신망법 제50조) — 열람 전용.

## 관리자 화면

`/admin/new-businesses` ([NewBusinessController](../app/Http/Controllers/Admin/NewBusinessController.php) · [new-businesses.blade.php](../resources/views/admin/new-businesses.blade.php))
- 화면 상단 법적 고지 배너는 **2026-07-17 운영 요청으로 제거**했다(되살리지 말 것). 발송 금지 원칙 자체는 유효 — 컨트롤러·모델 주석과 이 문서에 남겼다.
- 현황(플레이스 있음/미등록/미확인 클릭 필터 — 배지를 누르면 그 상태로 필터되고 현황 수치도 그 필터 기준이 된다) · **지금 수집**(≤7일 → 끝나면 플레이스 확인으로 이어짐) · **확인 실행(N)** / **전체 재확인(M)** — 버튼에 대상 수를 띄우고 **건수 입력은 없다(제한 없음)**. 대상이 0 이면 해당 버튼은 disabled.
- **실행은 fetch(JSON) + 진행률**(`form[data-nb-run]` submit 가로채기): 수집 1콜 → `place-match` 배치를 `remaining === 0` 까지 반복 → 진행률 카드(`#nb-progress`: 스피너·"플레이스 확인 중… 있음 N · 미등록 M"·N/총·바)로 표시 → 완료 요약을 `sessionStorage['nb-flash']` 에 넣고 새로고침해 목록·현황을 갱신. 실행 중엔 두 버튼 모두 잠근다(`pointer-events:none` — `disabled` 는 제출을 취소시킨다).
- **중단**(`#nb-stop`, 실행 중에만 보임): **진행 중인 배치는 끝내고 다음 배치부터 멈춘다** — 이미 네이버에 물어본 건의 결과를 버리지 않기 위해서다(그래서 누른 뒤 최대 배치 1회분 ≈ 5초 더 돈다). 수집 단계에서 누르면 이어지는 플레이스 확인을 시작하지 않는다(받아둔 수집분은 이미 저장됨). 결과는 "중단됨 — … · N건 남음" 으로 표시하고, 남은 건은 다음 실행이 이어서 처리한다. 실측: 194건 재확인 중 중단 → 15건 처리·179건 남김.
- **최종 확인 열** — `place_checked_at`(`m-d H:i`, 툴팁은 전체 일시). 미등록은 "그 시점에 없었다"는 뜻이라 언제 본 결과인지가 중요하다.
- JS 가 없으면 폼이 그대로 POST 되어 **배치 1회만** 처리하고 남은 수를 플래시로 안내한다(타임아웃 방지 폴백).
- 필터 한 줄: 기간 · 시/도 › 시/군/구 · 업종 · 상호 검색 (키워드 탐색과 동일 방식)
- 목록: 인허가일 · 상호 · 업종·업태 · 지역 · 주소 · **전화**(인허가 번호 없으면 플레이스 번호 + `안심`/`플레이스` 라벨) · **플레이스**(있음=지도 링크 / 미등록=지도검색 링크) · **최종 확인**
- 열람 로그: `newbiz: admin viewed`(사용자·조건) — 개인정보 접근 이력
